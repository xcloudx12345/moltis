# Plan: Add Nostr DM Channel Support

**Issue:** [moltis-org/moltis#668](https://github.com/moltis-org/moltis/issues/668)

## Context

Moltis supports six messaging channels (Telegram, Discord, Slack, MS Teams, WhatsApp, Matrix) but none are decentralized or censorship-resistant. Users who rely on Nostr have no native path to connect. The openclaw project (TypeScript) already ships a production Nostr channel (~8K LOC) with NIP-01/NIP-04 support, relay health tracking, circuit breakers, OTP/allowlist access control, and profile management. This plan ports the concept to Rust using `nostr-sdk` v0.44 (tokio-native, NIP-04 + NIP-44 built-in).

## Scope — What "comparable with other channels" means

To reach parity with the minimum bar from the channel integration checklist:

| Capability | Status | Notes |
|---|---|---|
| Inbound DMs | Must-have | NIP-04 encrypted DMs (kind:4), NIP-44 when sender supports it |
| Outbound text replies | Must-have | Encrypted reply to sender pubkey |
| Streaming responses | Must-have | Edit-in-place not possible on Nostr; stream internally, send final |
| DM policy + allowlist | Must-have | `open`, `allowlist`, `pairing`, `disabled` — by pubkey |
| OTP approval flow | Must-have | Unknown senders get a pairing code DM |
| Typed config + redacted export | Must-have | `NostrAccountConfig` struct |
| Web UI add/edit flows | Must-have | Settings page + advanced JSON editor |
| Config template + docs | Must-have | Same PR |
| Relay health + reconnect | Must-have | `nostr-sdk` Client handles reconnect; add circuit-breaker metrics |
| NIP-01 profile (kind:0) | Nice-to-have | Publish bot profile metadata on start |
| NIP-65 relay list | Nice-to-have | Future — advertise preferred relays |
| Media attachments | Nice-to-have | Future — NIP-94 file metadata |
| Reactions (NIP-25) | Nice-to-have | Future |

```rust
    pub secret_key: Secret<String>,     // nsec1... or 64-char hex
```

### Step 3: Event bus / subscription loop (`bus.rs`)

Core inbound pipeline (mirrors openclaw's `nostr-bus.ts`):

```
1. nostr-sdk Client subscribes: Filter { kinds: [4], p: [bot_pubkey], since }
2. Event received →
3. Dedup check (LRU seen tracker by event ID)
4. Skip self-messages (event.pubkey == bot_pubkey)
5. Skip stale events (created_at < since)
6. Verify event signature (nostr-sdk does this, but belt-and-suspenders)
7. Authorize sender (allowlist / OTP check) — BEFORE decrypt
8. Decrypt content (NIP-04 or NIP-44 based on event structure)
9. Size validation on plaintext
10. Dispatch to gateway via ChannelEventSink::dispatch_to_chat()
```

The bus runs as a `tokio::spawn` task per account. Cancellation via `CancellationToken`.

### Step 4: Outbound (`outbound.rs`)

Implement `ChannelOutbound`:
- `send_text(account_id, to_pubkey, text, reply_to)`:
  - Look up sender's secret key
  - Encrypt with NIP-04 (broad compat) or NIP-44
  - Build kind:4 event with p-tag
  - Publish via `client.send_event()`
- Error classification: relay failures → retryable; key/encrypt errors → not retryable

### Step 5: Access control (`access.rs`)

Reuse `moltis_channels::gating::{DmPolicy, check_dm_access}` patterns:
- `allowlist` — only allowed pubkeys
- `pairing` — unknown senders get an OTP code via DM, approved in web UI
- `open` — anyone can DM
- `disabled` — ignore all inbound

Implement `ChannelOtpProvider` for the OTP flow.

### Step 6: Plugin lifecycle (`plugin.rs`)

```rust
impl ChannelPlugin for NostrPlugin {
    async fn start_account(id, config) → start Client, connect relays, spawn bus loop
    async fn stop_account(id)           → cancel bus loop, disconnect
    async fn has_account(id) → bool
    async fn account_ids() → Vec<String>
    fn channel_type() → ChannelType::Nostr
    fn outbound() → NostrOutbound
    fn status() → NostrStatus (probe relays)
    fn as_otp_provider() → Some(self)
}
```

## NIP-04 vs NIP-44 Decision

Start with **NIP-04** (kind:4) for broad compatibility — it's what most Nostr clients use today for DMs. Add NIP-44 decryption support for inbound (detect by content format) so the bot can receive NIP-44 encrypted messages. Outbound defaults to NIP-04 for now with a config option to prefer NIP-44.

NIP-17 (gift-wrapped DMs) is a future enhancement — it requires more complex event handling and is less widely supported.

# Greploop

Iteratively fix a PR/MR/CL until Greptile gives a perfect review: 5/5 confidence, zero unresolved comments.

## Inputs

- **PR/MR/CL number** (optional): If not provided, detect the PR/MR for the current branch, or the default pending changelist for p4.

## Instructions

### 0. Detect platform

First check for Perforce, then fall back to git remote detection:

```bash
# Check for Perforce environment
if p4 info >/dev/null 2>&1; then
  VCS="perforce"
else
  REMOTE_URL=$(git remote get-url origin)
  if echo "$REMOTE_URL" | grep -qi "gitlab"; then
    VCS="gitlab"
  else
    VCS="github"
  fi
fi
```

For self-hosted GitLab instances whose hostname doesn't contain "gitlab", the user can override by passing `--vcs gitlab` as an input. For Perforce, pass `--vcs perforce`.

### 1. Identify the PR/MR/CL

**GitHub:**
```bash
gh pr view --json number,headRefName -q '{number: .number, branch: .headRefName}'
```

**GitLab:**
```bash
glab mr view --output json | jq '{iid: .iid, branch: .source_branch}'
```

Switch to the PR/MR branch if not already on it.

**Perforce:**
```bash
# List pending changelists for current user/client
p4 changes -s pending -u $P4USER -c $P4CLIENT

# Describe a specific CL
p4 describe -s <CL_NUMBER>
```

Ensure the correct workspace (`p4 client`) is set before proceeding.

Key field differences:
- GitHub: `number`, `headRefName`, `headRefOid`
- GitLab: `iid`, `source_branch`, `sha`
- Perforce: changelist number, `P4CLIENT`, shelved files

### 2. Loop

Repeat the following cycle. **Max 5 iterations** to avoid runaway loops.

#### A. Trigger Greptile review

Push/shelve the latest changes (if any):

**GitHub/GitLab:**
```bash
git push
```

**Perforce:**
```bash
# Re-shelve to update the shelved files for review
p4 shelve -f -c <CL_NUMBER>
```

Wait for checks to start after push/shelve:

```bash
sleep 5
```

**GitHub** — check if Greptile is already running before posting a new trigger comment:

```bash
GREPTILE_STATE=$(gh pr checks <PR_NUMBER> --json name,state | jq -r '.[] | select(.name | test("greptile"; "i")) | .state')
```

If Greptile is **not** already running (`PENDING` or `IN_PROGRESS`), request a fresh review:

```bash
if [ "$GREPTILE_STATE" != "PENDING" ] && [ "$GREPTILE_STATE" != "IN_PROGRESS" ]; then
  gh pr comment <PR_NUMBER> --body "@greptile review"
fi
```

Then poll for the Greptile check run to complete:

```bash
HEAD_SHA=$(gh pr view <PR_NUMBER> --json headRefOid -q .headRefOid)

while true; do
  GREPTILE_CHECK=$(gh api "repos/{owner}/{repo}/commits/$HEAD_SHA/check-runs" \
    --jq '.check_runs[] | select(.name | test("greptile"; "i"))' 2>/dev/null)
  
  if [ -z "$GREPTILE_CHECK" ]; then
    echo "Waiting for Greptile check to appear..."
    sleep 5
    continue
  fi
  
  STATUS=$(echo "$GREPTILE_CHECK" | jq -r '.status // "completed"')
  CONCLUSION=$(echo "$GREPTILE_CHECK" | jq -r '.conclusion // "pending"')
  
  if [ "$STATUS" = "completed" ]; then
    if [ "$CONCLUSION" = "success" ]; then
      echo "Greptile check passed!"
    else
      echo "Greptile check completed with: $CONCLUSION"
    fi
    break
  fi
```
