# Fix: GitHub Copilot provider — Responses API support for gpt-5.4+

## Context

GitHub Copilot provider (`crates/providers/src/github_copilot.rs`) hardcodes `/chat/completions` for all models. Newer OpenAI models (gpt-5.4, gpt-5.4-pro, gpt-5.2-pro) only support the Responses API (`/responses`), returning HTTP 400 with `unsupported_api_for_model`. See issue #392.

The OpenAI provider already has Responses API support via WebSocket, but that's gated behind `api.openai.com` domain checks. The Copilot provider needs its own Responses API path using HTTP+SSE (simpler, no WebSocket pool needed).

## Approach

Model-based routing with error-based fallback: dispatch to `/responses` for known models, and detect `unsupported_api_for_model` errors for unknown future models.

## Files to modify

### 1. `crates/providers/src/openai_compat.rs` — shared utilities

- **Move `split_responses_instructions_and_input()`** from `OpenAiProvider` impl to a free function here. It has no `self` dependency — just splits system messages into `instructions` and passes the rest through `to_responses_input()`.

- **Add `responses_output_index()`** — extracted from `ws_output_index()` in `openai.rs` (resolves `output_index` / `item_index` / `index` from a Responses API event). Both providers can share it.

- **Add `process_responses_sse_line()`** — new function that processes a single SSE `data:` line from a Responses API stream. Returns `SseLineResult` (reusing existing enum). Handles event types:
  - `response.output_text.delta` → `StreamEvent::Delta`
  - `response.output_item.added` (type=function_call) → `StreamEvent::ToolCallStart`
  - `response.function_call_arguments.delta` → `StreamEvent::ToolCallArgumentsDelta`
  - `response.function_call_arguments.done` → `StreamEvent::ToolCallComplete`
  - `response.completed` → parse usage, return `SseLineResult::Done`
  - `error` / `response.failed` → `StreamEvent::Error`

- **Add `ResponsesStreamState`** — state struct for Responses API SSE streaming (tool_calls map, completed_tool_calls set, current_tool_index, usage tokens). Similar to the local variables in `stream_with_tools_websocket()`.

- **Add `finalize_responses_stream()`** — like `finalize_stream()` but for Responses API state (emits pending ToolCallComplete + Done with usage).

- **Add `parse_responses_completion()`** — parse a non-streaming Responses API JSON response into `CompletionResponse`. Handles `output` array with `message` and `function_call` items.

### 3. `crates/providers/src/github_copilot.rs` — add Responses API

**Add `needs_responses_api(model: &str) -> bool`:**
```rust
fn needs_responses_api(model: &str) -> bool {
    let m = model.to_ascii_lowercase();
    m.starts_with("gpt-5.4")
        || m == "gpt-5.2-pro"
        || m.starts_with("codex-")
}
```

**Add `is_responses_api_required_error(body: &str) -> bool`:**
Check for `"unsupported_api_for_model"` or `"not accessible via the /chat/completions"` in the error response.

**Add `complete_responses()`:**
- POST to `{COPILOT_API_BASE}/responses` (non-streaming)
- Build request body using `split_responses_instructions_and_input()`, `to_responses_api_tools()`
- Parse response using `parse_responses_completion()`

**Modify `complete()`:**
- If `needs_responses_api(&self.model)`, dispatch directly to `complete_responses()`
- Otherwise, try chat completions. On 400 error with `is_responses_api_required_error()`, retry with `complete_responses()`

**Add `stream_responses_api()`:**
- POST to `{COPILOT_API_BASE}/responses` with `"stream": true`
- Build request body same as `complete_responses()` but with stream options
- Parse SSE events using `process_responses_sse_line()` + `ResponsesStreamState`
- Follow existing SSE parsing pattern (byte_stream → buf → line splitting → event processing)

**Modify `stream_with_tools()`:**
- If `needs_responses_api(&self.model)`, dispatch to `stream_responses_api()`
- Otherwise, use existing chat completions streaming
- Add error detection: if 400 with `is_responses_api_required_error()`, redirect to `stream_responses_api()` (detected before streaming starts, in the error_for_status_ref check)

**Update `COPILOT_MODELS`:**
- Add `("gpt-5.4", "GPT-5.4 (Copilot)")`, `("gpt-5.4-pro", "GPT-5.4 Pro (Copilot)")`, `("gpt-5.2-pro", "GPT-5.2 Pro (Copilot)")`
