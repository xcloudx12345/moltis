# Plan: Matrix OIDC Authentication (Issue #711)

## Context

Matrix password authentication is increasingly rare. Modern homeservers (including matrix.org since April 2025) use Matrix Authentication Service (MAS), which implements OAuth 2.0/OIDC via MSC3861. Users on these homeservers cannot connect Moltis because only `password` and `access_token` auth modes exist today. This is blocking for users like the issue reporter.

matrix-sdk 0.16 (already in use) has a built-in OAuth/OIDC API (`client.oauth()`) that requires no feature flags. It handles PKCE, dynamic client registration, token exchange, and automatic token refresh.

## Design Decision: Two-Phase Browser Flow

OIDC requires browser-based authentication. The flow mirrors the existing MCP/provider OAuth pattern:

1. **Start** тАФ Backend discovers OIDC metadata, registers client, returns auth URL
2. **Complete** тАФ User authenticates in browser, provider redirects to `/auth/callback`, backend exchanges code for tokens

This reuses the existing `/auth/callback` handler in `crates/web/src/oauth.rs` with a third fallback for channel OIDC.

- Update `RedactedConfig`, `Debug` impl, `Default` impl, tests
- Keep full backward compatibility: existing configs with `password`/`access_token` and no `auth_mode` field work identically

### Step 2: Auth mode dispatch (`crates/matrix/src/client.rs`)

- Extend `AuthMode` enum with `Oidc` variant
- Update `auth_mode()`: check explicit `config.auth_mode` first, then fall back to existing credential sniffing
- Update `authenticate_client()`: add `AuthMode::Oidc` arm that calls `oidc::restore_oidc_session()`; return clear error if no saved session exists (OIDC requires the two-phase RPC flow first)
- Update `build_client()`: accept auth mode parameter; when OIDC, enable `handle_refresh_tokens(true)` on `ClientBuilder`

### Step 3: OIDC module (NEW: `crates/matrix/src/oidc.rs`)

Core OIDC logic, keeping `client.rs` under 1500 lines:

```
pub struct OidcLoginPending { pub auth_url: String, pub state: String }

// Session persistence
fn oidc_session_path(account_id) -> PathBuf
async fn save_oidc_session(account_id, session: &OAuthSession) -> ChannelResult<()>
async fn load_oidc_session(account_id) -> ChannelResult<Option<OAuthSession>>

// OIDC flow
fn build_client_metadata(redirect_uri) -> ClientMetadata
async fn start_oidc_login(client, account_id, config, redirect_uri) -> ChannelResult<OidcLoginPending>
async fn finish_oidc_login(client, account_id, callback_url) -> ChannelResult<AuthenticatedMatrixAccount>
async fn restore_oidc_session(client, account_id) -> ChannelResult<AuthenticatedMatrixAccount>

// Token refresh persistence
fn spawn_session_persistence_task(client, account_id) // subscribes to session changes
```

Key details:
- Uses `client.oauth().server_metadata()` for discovery
- Uses `client.oauth().register_client()` for dynamic registration
- Uses `client.oauth().login().build()` to generate auth URL with PKCE
- Uses `client.oauth().finish_login()` to exchange code for tokens
- Spawns background task via `client.subscribe_to_session_changes()` to persist refreshed tokens
- Cross-signing: uses `bootstrap_cross_signing_if_needed(None)` instead of password-based UIAA; handles `CrossSigningResetAuthType::OAuth` (already handled in existing code at `client.rs:428`)

### Step 4: Plugin OIDC methods (`crates/matrix/src/plugin.rs`)

Add to `MatrixPlugin`:
- `oidc_start(account_id, homeserver, redirect_uri, config) -> Result<Value>` тАФ builds client, starts OIDC flow, stores pending state in `AccountStateMap`
- `oidc_complete(state, callback_url) -> Result<Value>` тАФ finds pending state by CSRF token, finishes login, moves client to active state

Add to `AccountState` (`crates/matrix/src/state.rs`):
- `oidc_pending: Mutex<Option<OidcPendingState>>` тАФ holds pre-auth `Client` + CSRF state + config between start/complete

