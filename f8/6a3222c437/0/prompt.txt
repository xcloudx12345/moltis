# Plan: Add Nostr DM Channel Support

**Issue:** [moltis-org/moltis#668](https://github.com/moltis-org/moltis/issues/668)

## Context

Moltis supports six messaging channels (Telegram, Discord, Slack, MS Teams, WhatsApp, Matrix) but none are decentralized or censorship-resistant. Users who rely on Nostr have no native path to connect. The openclaw project (TypeScript) already ships a production Nostr channel (~8K LOC) with NIP-01/NIP-04 support, relay health tracking, circuit breakers, OTP/allowlist access control, and profile management. This plan ports the concept to Rust using `nostr-sdk` v0.44 (tokio-native, NIP-04 + NIP-44 built-in).

## Scope вЂ” What "comparable with other channels" means

To reach parity with the minimum bar from the channel integration checklist:

| Capability | Status | Notes |
|---|---|---|
| Inbound DMs | Must-have | NIP-04 encrypted DMs (kind:4), NIP-44 when sender supports it |
| Outbound text replies | Must-have | Encrypted reply to sender pubkey |
| Streaming responses | Must-have | Edit-in-place not possible on Nostr; stream internally, send final |
| DM policy + allowlist | Must-have | `open`, `allowlist`, `pairing`, `disabled` вЂ” by pubkey |
| OTP approval flow | Must-have | Unknown senders get a pairing code DM |
| Typed config + redacted export | Must-have | `NostrAccountConfig` struct |
| Web UI add/edit flows | Must-have | Settings page + advanced JSON editor |
| Config template + docs | Must-have | Same PR |
| Relay health + reconnect | Must-have | `nostr-sdk` Client handles reconnect; add circuit-breaker metrics |
| NIP-01 profile (kind:0) | Nice-to-have | Publish bot profile metadata on start |
| NIP-65 relay list | Nice-to-have | Future вЂ” advertise preferred relays |
| Media attachments | Nice-to-have | Future вЂ” NIP-94 file metadata |
| Reactions (NIP-25) | Nice-to-have | Future |

### Step 3: Event bus / subscription loop (`bus.rs`)

Core inbound pipeline (mirrors openclaw's `nostr-bus.ts`):

```
1. nostr-sdk Client subscribes: Filter { kinds: [4], p: [bot_pubkey], since }
2. Event received в†’
3. Dedup check (LRU seen tracker by event ID)
4. Skip self-messages (event.pubkey == bot_pubkey)
5. Skip stale events (created_at < since)
6. Verify event signature (nostr-sdk does this, but belt-and-suspenders)
7. Authorize sender (allowlist / OTP check) вЂ” BEFORE decrypt
8. Decrypt content (NIP-04 or NIP-44 based on event structure)
9. Size validation on plaintext
10. Dispatch to gateway via ChannelEventSink::dispatch_to_chat()
```

The bus runs as a `tokio::spawn` task per account. Cancellation via `CancellationToken`.

### Step 4: Outbound (`outbound.rs`)

Implement `ChannelOutbound`:
- `send_text(account_id, to_pubkey, text, reply_to)`:
  - Look up sender's secret key
  - Encrypt with NIP-04 (broad compat) or NIP-44
  - Build kind:4 event with p-tag
  - Publish via `client.send_event()`
- Error classification: relay failures в†’ retryable; key/encrypt errors в†’ not retryable

### Step 5: Access control (`access.rs`)

Reuse `moltis_channels::gating::{DmPolicy, check_dm_access}` patterns:
- `allowlist` вЂ” only allowed pubkeys
- `pairing` вЂ” unknown senders get an OTP code via DM, approved in web UI
- `open` вЂ” anyone can DM
- `disabled` вЂ” ignore all inbound

Implement `ChannelOtpProvider` for the OTP flow.

## NIP-04 vs NIP-44 Decision

Start with **NIP-04** (kind:4) for broad compatibility вЂ” it's what most Nostr clients use today for DMs. Add NIP-44 decryption support for inbound (detect by content format) so the bot can receive NIP-44 encrypted messages. Outbound defaults to NIP-04 for now with a config option to prefer NIP-44.

NIP-17 (gift-wrapped DMs) is a future enhancement вЂ” it requires more complex event handling and is less widely supported.
