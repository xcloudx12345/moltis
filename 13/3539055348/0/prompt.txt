# Plan: Redacted serialization path for channel config API responses

Issue: https://github.com/moltis-org/moltis/issues/462

## Context

`account_config_json()` on each channel plugin serializes config structs via `serde_json::to_value()`, which uses the storage serialization path (`serialize_secret`) that exposes raw secret values. This means API responses to the frontend leak tokens and keys. PR #449's approach of hand-rolling a `redact_channel_config()` walking raw JSON was rejected as fragile.

The fix uses **Option 1 from the issue**: a `Redacted<T>` wrapper type that serializes `Secret<String>` fields as `"[REDACTED]"`, used only by `account_config_json()`.

## Plan

### Step 1: Add shared secret serde helpers to `moltis-common`

**Files:**
- `crates/common/Cargo.toml` — add `secrecy = { workspace = true }`
- `crates/common/src/lib.rs` — add `pub mod secret_serde;`
- `crates/common/src/secret_serde.rs` — new file

The new module provides four public functions:
- `serialize_secret` — exposes raw value (for storage)
- `serialize_option_secret` — exposes raw optional value (for storage)
- `serialize_secret_redacted` — writes `"[REDACTED]"` (for API)
- `serialize_option_secret_redacted` — writes `"[REDACTED]"` or `null` (for API)

### Step 2: Create `Redacted<T>` wrapper type

In `crates/common/src/secret_serde.rs`, add:

```rust
/// Wrapper that serializes `Secret<String>` fields as `"[REDACTED]"`.
/// Use for API responses where secrets must not leak.
pub struct Redacted<'a, T>(pub &'a T);
```

Each channel config struct will get a companion `impl Serialize for Redacted<'_, FooConfig>` that delegates to a shadow struct using `serialize_secret_redacted` instead of `serialize_secret`. This is the cleanest approach because:
- No cloning of the config struct
- No runtime string matching on field names
- Compile-time guarantee that all secret fields are redacted
- Storage serialization path is unchanged

### Step 3: Add `Redacted` Serialize impls to each channel config

**Files to modify:**

| File | Struct | Secret fields |
|------|--------|--------------|
| `crates/discord/src/config.rs` | `DiscordAccountConfig` | `token` |
| `crates/telegram/src/config.rs` | `TelegramAccountConfig` | `token` |
| `crates/slack/src/config.rs` | `SlackAccountConfig` | `bot_token`, `app_token`, `signing_secret` |
| `crates/msteams/src/config.rs` | `MsTeamsAccountConfig` | `app_password`, `webhook_secret` |

WhatsApp has no secret fields — no changes needed.

For each config, implement `Serialize for Redacted<'_, XxxAccountConfig>` that serializes every field identically to the normal `Serialize` impl except secret fields use `serialize_secret_redacted` / `serialize_option_secret_redacted`.

**Approach**: Use a helper macro or manual impl with `SerializeStruct`. Since each config has different fields, manual impl is clearer and avoids macro complexity.
