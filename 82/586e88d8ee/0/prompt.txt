# Plan: Auto-Continue on Model Stop + Max Iterations UX

## Context

A user reported that Moltis "just stops" mid-task with no explanation. Their log showed:
```
agent run complete iterations=12 tool_calls=11
```

**Diagnosis:** The log says `"agent run complete"` (success path), not `"agent loop exceeded max iterations"` (error path). The model chose to stop at iteration 12 — it returned text without tool calls. The default max is 25, so there was budget remaining.

**Two problems to solve:**
1. **Model stops naturally mid-task** — the model returns text without tool calls while it still has iteration budget. Currently: silent, no explanation, no recovery. **Fix: auto-continue in the agent loop** and show a notice to the user.
2. **Max iterations actually hit** — falls through to generic "unknown" error. **Fix: proper error type with Continue button.**

---

## Part A: Auto-Continue When Model Stops Mid-Task

### A1. Add `AutoContinue` variant to `RunnerEvent` (`crates/agents/src/runner.rs:464`)

Add a new event variant so the chat service can notify the UI:

```rust
/// The model stopped without tool calls but iteration budget remains;
/// the runner is automatically re-prompting.
AutoContinue {
    iteration: usize,
    max_iterations: usize,
},
```

### A2. Auto-continue logic in the agent loop (`crates/agents/src/runner.rs`)

At the break point where `tool_calls.is_empty()` (streaming: ~line 1797, sync: ~line 1126):

**Before** returning `Ok(AgentRunResult { ... })`, add this check:

```rust
if tool_calls.is_empty() {
    // Auto-continue: if the model made tool calls earlier in this run
    // and we haven't exhausted iterations, nudge it to keep going.
    if total_tool_calls > 0 && auto_continue_count < MAX_AUTO_CONTINUES {
        auto_continue_count += 1;
        info!(
            iterations,
            auto_continue_count,
            "model stopped without tool calls, auto-continuing"
        );
        if let Some(cb) = on_event {
            cb(RunnerEvent::AutoContinue {
                iteration: iterations,
                max_iterations,
            });
        }
        // Append the model's response + a nudge as a user message
        let response_text = /* accumulated_text or response.text */;
        if !response_text.is_empty() {
            messages.push(ChatMessage::assistant(response_text));
        }
        messages.push(ChatMessage::user("Continue with the task. If you've completed it, summarize your results."));
        continue; // re-enter the loop
    }

    // Original return logic...
    return Ok(AgentRunResult { ... });
}
```

**Key design decisions:**
- `MAX_AUTO_CONTINUES = 2` — hard cap to prevent infinite loops. The model gets 2 nudges; if it still stops, we accept it.
- Only triggers when `total_tool_calls > 0` — pure Q&A (no tool use) doesn't auto-continue.
- The nudge message tells the model to continue OR summarize, so if it's genuinely done it can say so cleanly.
- Add `auto_continue_count` variable initialized to 0 at the top of the loop, alongside `iterations`.

Both the streaming and sync loops need this change (lines ~1797 and ~1126).

### A3. Map `AutoContinue` event to a notice broadcast (`crates/chat/src/lib.rs`)

In the `RunnerEvent` match in the event callback (around line 6126), add:

```rust
RunnerEvent::AutoContinue { iteration, max_iterations } => serde_json::json!({
    "runId": run_id,
    "sessionKey": sk,
    "state": "notice",
    "title": "Auto-continue",
    "message": format!("Model paused at iteration {}/{}. Asking it to continue...", iteration, max_iterations),
    "seq": seq,
}),
```

This reuses the existing `handleChatNotice` handler on the frontend — no new JS needed for the notice. The user sees a system message like: **Auto-continue:** Model paused at iteration 12/25. Asking it to continue...

### A4. Channel compatibility (no extra work needed)

**Auto-continue is transparent to channels.** The logic runs inside `runner.rs` before the loop exits, so Telegram/Matrix/Discord users simply receive the final response after any auto-continues complete. The `AutoContinue` notice broadcasts to WebSocket for real-time web UI visibility, but channel users don't need it — they just see the result.
