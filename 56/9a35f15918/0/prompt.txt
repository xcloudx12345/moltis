# Plan: Migrate to Date-Based Release Versioning (YYYYMMDD.NN)

## Context

Migrate moltis from semver (`v0.10.18`) to date-based versioning (`20260311.01`) matching the arbor project pattern. The user-facing version becomes `YYYYMMDD.NN` (date + daily sequence number). Cargo.toml stays at a static `0.1.0` since Cargo enforces semver, and the real version is injected at build time via `MOLTIS_VERSION` env var.

## 2. Update Checker (`update_check.rs`)

**File:** `crates/gateway/src/update_check.rs`

**Changes:**
- Replace `parse_semver_triplet()` with `parse_date_version()` → returns `Option<(u32, u32)>` for `(YYYYMMDD, NN)`
- `is_newer_version()` — compare `(date, seq)` tuples (natural ordering works)
- `is_pre_release()` — remove or repurpose. Date versions don't have pre-release suffixes. Could check for `-` suffix for future use, or remove the unstable channel logic entirely
- `normalize_version()` — keep, still strips leading `v`/`V` and whitespace
- Update `releases.json` manifest: the `unstable` channel concept may not apply anymore. Keep the struct for backward compat but `is_pre_release` effectively always returns false for date versions
- Update all tests to use date-based versions

**Key concern:** Users on old semver builds (0.10.18) checking against a new date version (20260311.01) — the parser must handle both formats gracefully. `parse_date_version("0.10.18")` should return `None`, and `parse_semver_triplet("20260311.01")` should also return `None`. We need a `is_newer_version` that tries date format first, then semver, and considers any date version newer than any semver version. This ensures old users see the update.

## Backward Compatibility / Migration

### Critical: Update alerts for existing users
Users running `0.10.18` will check `releases.json` and get back a date version like `20260311.01`. The update checker MUST:
1. Try date-version parsing first
2. Fall back to semver parsing
3. If formats differ (current=semver, latest=date), treat the date version as newer
4. This ensures all existing users see the update notification

### Tag history
Old `v*` tags remain in git history. New tags use `YYYYMMDD.NN` format. Both coexist.

### Homebrew
Users on old `brew install` will get the new version naturally — the formula just points to new URLs.

### Docker
Old image tags (`0.10.18`, `0.10`) remain. New images get `20260311.01`, `sha-*`, `latest`.
