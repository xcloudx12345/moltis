# Plan: Fix Podman sandbox startup failure (#757)

## Context

PR #706 added sysfs tmpfs overlays to `hardening_args()` in `docker.rs` to mask host hardware identifiers (serial numbers, BIOS data, disk models, LUKS UUIDs). These work on Docker but break Podman on Ubuntu 24.04/26.04 with both runc and crun.

**Root cause:** When Podman mounts a tmpfs over sysfs paths like `/sys/devices/virtual/dmi`, the OCI runtime performs "tmpcopyup" — copying existing directory contents into the new tmpfs. Some sysfs files (e.g., `board_serial`) are permission-denied even for root inside the container (because `--cap-drop ALL` is active), causing the mount to fail entirely.

Docker does not exhibit this behavior because it handles tmpfs overlay differently on sysfs paths.

## Approach

Skip the sysfs tmpfs overlays when the backend is Podman. Podman's OCI runtime already masks `/sys/firmware` by default via its built-in `MaskedPaths`. For the other paths (`/sys/class/dmi`, `/sys/devices/virtual/dmi`, `/sys/class/block`), losing the masking is acceptable — a slightly less hardened but functional sandbox is better than a broken one.

No config option needed for the initial fix — the backend is already known at runtime (`self.cli`). If users report similar issues with specific Docker configurations later, a config escape hatch can be added as follow-up.

## Changes

### 1. `crates/tools/src/sandbox/docker.rs`

- Change `hardening_args(is_prebuilt: bool)` signature to `hardening_args(is_prebuilt: bool, cli: &str)`
- Wrap the sysfs tmpfs overlays (lines 203-210) in a `cli != "podman"` guard
- Keep `--hostname sandbox` for all backends (this works on Podman)
- Update the call site at line 324 to pass `self.cli`

### 2. `crates/tools/src/sandbox/tests/core.rs`

### 3. `docs/src/sandbox.md`

- Add a note that sysfs tmpfs masking is Docker-only; Podman relies on its built-in OCI `MaskedPaths` for `/sys/firmware` masking
