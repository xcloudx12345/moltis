# Fix #796: Sandbox image not exported to Podman store after BuildKit build

## Context

**Issue:** [#796](https://github.com/moltis-org/moltis/issues/796)
**Environment:** Podman 5.4.2, Debian 13, `backend = "auto"`

When Podman delegates `podman build` to BuildKit (via a `buildx_buildkit_default` container),
the build exits 0 but the image lands in BuildKit's internal cache — not in the Podman store.
Subsequent `podman image inspect` / `podman run` fail with "image not found".

The code at `docker.rs:build_image` only checks the exit code, never verifies the image
actually exists in the local store after building.

## Approach

**Post-build verification + export retry** in `DockerSandbox::build_image()`.

1. After `podman build` succeeds, verify the image is in the store via `sandbox_image_exists()`
2. If missing (BuildKit scenario), re-run the build with `--output type=docker,dest=<tmpfile>`
   and pipe to `podman load -i <tmpfile>` to force the image into the Podman store
3. BuildKit's layer cache makes the re-build essentially free (same Dockerfile = cache hit)
4. Normal Podman+buildah flows are unaffected — the verify check passes immediately

Why this over adding `--load` to the initial build: Podman's `build` command doesn't support
`--load` (that's Docker buildx–only). The `--output type=docker,dest=<file>` + `podman load`
pattern is portable across all Podman 4.x+ versions.
