# External CLI Agent Bridge for Moltis

## Context

Moltis currently runs its own agent loop (LLM provider + tool execution) for all
chat sessions. The Polyphony project at `~/code/polyphony` demonstrates a
pluggable architecture where CLI coding agents (Claude Code, opencode, Codex CLI,
Pi agent) are dispatched as external backends via PTY/tmux sessions and JSON-RPC
protocols, with context snapshots passed between orchestrator and agent.

**Goal**: Let Moltis chat sessions connect to external CLI agents. The CLI agent
owns its own session state (memory, context window). Moltis acts as orchestrator
and source of truth for conversation history. On session reattach, only delta
(new messages) is sent. Compaction mismatches between Moltis and the CLI agent
are handled gracefully.

---

## Architecture Overview

```
User → WebSocket → Gateway → LiveChatService.send()
                                     │
                        ┌────────────┴────────────┐
                        │                         │
                  (no binding)            (external agent bound)
                        │                         │
                 run_agent_loop()         SessionBridge
                        │                    │        │
                   LLM Provider        CliTransport  AcpTransport
                                       (PTY/tmux)   (JSON-RPC)
                                           │              │
                                     claude/opencode  ACP agents
                                     codex/pi
```

## Session Bridge Protocol

### BridgeState (persisted per session in SQLite KV)

```rust
pub struct BridgeState {
    pub agent_kind: AgentTransportKind,      // "claude-code", "codex", etc.
    pub external_session_id: Option<String>, // CLI agent's own session ID
    pub last_synced_message_index: u32,      // Moltis message count at last sync
    pub last_synced_tail_hash: Option<String>,// SHA-256 of message at sync point
    pub initialized: bool,                   // Has full context been sent?
    pub compaction_generation: u64,          // Incremented on compaction detection
}
```

### Delta Sync Flow

1. **First turn**: Send full `ContextSnapshot` (system prompt, project context,
   memory files) + user prompt. Mark `initialized = true`.

2. **Subsequent turns**:
   - Read Moltis session messages
   - If `messages.len() > last_synced_message_index`:
     - Hash message at `last_synced_message_index - 1`
     - If hash matches → send only delta (new messages since sync point)
     - If hash mismatch → compaction detected, fall to reconciliation
   - Update `last_synced_message_index` and `last_synced_tail_hash`

3. **Reattach** (session reconnection):
   - Check if external agent session is still alive
   - If alive → delta sync (steps above)
   - If dead → spawn new session, send context snapshot + recent messages

### Compaction Reconciliation

| Moltis compacted? | CLI compacted? | Action |
|---|---|---|
| No | No | Delta sync (normal) |
| Yes | No | CLI has more context. Send summary of compacted region + new messages. |
| No | Yes | CLI lost context. Re-send full context snapshot from Moltis history. |
| Yes | Yes | Both lost detail. Send Moltis's compacted summary as authoritative context. |

**Detection**:
- **Moltis compacted**: `tail_hash` mismatch + `messages.len() < last_synced_message_index`
- **CLI compacted**: Session not found, or CLI reports stale thread (agent-specific detection)

**Key insight from Polyphony**: Treat the CLI agent as *effectively stateless* from
Moltis's perspective. Always be prepared to re-send context. The `BridgeState`
sync cursor is an optimization, not a correctness requirement.

## Context Snapshot Format

```rust
pub struct ContextSnapshot {
    pub working_dir: Option<PathBuf>,
    pub summary: Option<String>,           // Compacted conversation summary
    pub recent_turns: Vec<ContextTurn>,    // Last N message pairs
    pub system_instructions: Option<String>,
    pub memory_files: HashMap<String, String>, // e.g., CLAUDE.md contents
    pub project_context: Option<String>,   // Project-specific context
}
```

Delivered to CLI agents via:
- **Env vars**: `MOLTIS_CONTEXT_FILE=<path>`, `MOLTIS_SESSION_KEY=<key>`
- **File**: JSON written to temp path
- **Stdin**: For agents accepting piped input (Claude Code `-p` mode)

## Per-Runtime Integration

### Claude Code (`runtimes/claude_code.rs`)

- **Binary**: `claude` (via `which::which`)
- **Two modes**:
  - `-p` (print/one-shot): Prompt via stdin, response on stdout. Best for single turns.
  - Interactive: Tmux session, prompt via tmux paste. Best for multi-turn with tool use.
  - `--resume <session-id>`: Reattach to existing Claude Code session (avoids resending context)
  - `--continue`: Continue last session in working directory
- **Session ID**: Claude Code's internal session ID, captured from output or `--output-format json`
- **Output parsing**: `--output-format json` gives structured JSON (in `-p` mode); interactive mode needs terminal output parsing via `vt100::Parser`
- **Memory**: Claude Code's `.claude/` directory is preserved (it's in the project working dir)

### opencode (`runtimes/opencode.rs`)

- **Binary**: `opencode`
- **Mode**: Interactive TUI. Tmux backend required.
- **Prompt delivery**: Tmux paste
- **Output**: Terminal output parsing

### Codex CLI (`runtimes/codex.rs`)

- **Binary**: `codex`
- **Mode**: JSON-RPC over stdio (structured, like Polyphony's implementation)
- **Protocol**: `{"method": "turn/start", "params": {...}}` request/response
- **Reuse**: `StdioTransport` pattern from `crates/mcp/src/transport.rs`

### Pi Agent (`runtimes/pi_agent.rs`)

- **Binary**: `pi`
- **Mode**: JSON-RPC with state management
- **Protocol**: `get_state`, `prompt`, `abort` methods
- **Events**: `message_update`, `message_end`, `tool_execution_start/end`

### ACP (`runtimes/acp.rs`)

- **Protocol**: JSON-RPC over stdio (Agent Communication Protocol)
- **Lifecycle**: `InitializeRequest` → `NewSessionRequest` → `PromptRequest` turns
- **Terminal support**: ACP agents can request terminal creation/output
- **Tool delegation**: ACP agents may request tool execution from Moltis (reverse tool calls via `ToolRegistry`)
- **Reuse**: `StdioTransport` from `crates/mcp/src/transport.rs` as base

## Gateway Integration

### Routing in `LiveChatService::send()` (crates/chat/src/lib.rs:2832)

After resolving `session_key` (line ~2906), add a routing check:

```rust
// After session_key resolution, before shell command check:
if let Some(bridge) = self.external_agent_bridge(&session_key).await {
    return self.send_via_external_agent(
        bridge, &session_key, &message_content, &params,
    ).await;
}
```

The `send_via_external_agent` method:
1. Persists user message to Moltis session store (same as normal flow)
2. Computes context delta via `SessionBridge`
3. Calls `ExternalAgentSession::send_prompt()`
4. Streams events back, broadcasting via existing WebSocket events (`chat:delta`, `chat:tool_start`, `chat:final`)
5. Persists assistant response to session store

**No changes to WebSocket protocol or web UI rendering** - external agent events map
to the same event types the UI already handles.
