# Fix: OpenClaw import overwrites config template comments (GH-458)

## Context

When a user does a fresh install and imports from OpenClaw, the commented config examples in `moltis.toml` are lost. This happens because:

1. Server starts тЖТ `discover_and_load()` тЖТ no config file тЖТ `write_default_config()` writes the **full template** with all commented examples (~700 lines of documented options)
2. User goes through onboarding тЖТ clicks "Import from OpenClaw"
3. `persist_identity()` and `persist_channels()` in `crates/openclaw-import/src/lib.rs` call a **local** `save_config_to_path()` (line 496) that does `toml::to_string_pretty(config)` тЖТ `fs::write()`, **overwriting** the template with a bare ~20-line config

The `moltis_config` crate already has a comment-preserving save path (`moltis_config::loader::save_config_to_path`, line 981) that uses `toml_edit` to merge new values into the existing document while preserving comments. The openclaw-import crate simply doesn't use it.

## Plan

**File: `crates/openclaw-import/src/lib.rs`**

1. **Replace the local `save_config_to_path`** (lines 496-503) тАФ instead of doing raw `toml::to_string_pretty()` + `fs::write()`, delegate to `moltis_config::loader::save_config_to_path` which preserves comments.

2. **Update `persist_identity`** (line 393) and **`persist_channels`** (line 447) тАФ these already call `save_config_to_path`. Once we replace the local function body to delegate to `moltis_config`, they will automatically benefit.

3. The simplest approach: rewrite the local `save_config_to_path` to call `moltis_config::loader::save_config_to_path` and map the error type. This keeps all call sites unchanged.

   ```rust
   fn save_config_to_path(path: &Path, config: &moltis_config::MoltisConfig) -> error::Result<()> {
       moltis_config::loader::save_config_to_path(path, config)?;
       Ok(())
   }
   ```

4. **Handle the "no file yet" edge case**: If the import runs before `discover_and_load()` creates the template (unlikely in normal flow but possible in tests), `moltis_config::loader::save_config_to_path` falls through to plain `fs::write()` when the file doesn't exist тАФ which is the same current behavior. No regression.

5. **Remove the now-unused `toml` direct dependency** from `crates/openclaw-import/Cargo.toml` if `toml::to_string_pretty` is no longer called anywhere in the crate. (Need to check if other modules still use it.)
