# Plan: Fix repeated sandbox package installs + add observability

## Context

GitHub discussion #781: User sees moltis continuously spawning `dpkg` processes
(Docker image builds / package installs) without any moltis logs explaining what's
happening or why.

**Root cause analysis**: There is no single infinite loop, but the architecture
creates repeated provisioning under failure conditions:

1. `ensure_ready()` is called on **every** tool exec (idempotent — returns early
   if container already running).
2. If the container dies between commands, the retry path in `exec.rs:634-659`
   calls `cleanup()` + `ensure_ready()` again, which runs the full pipeline:
   `resolve_local_image()` → possibly `build_image()` (docker build with apt-get)
   → `docker run` → `provision_packages()` (apt-get install inside container).
3. If the **startup pre-build fails**, every session falls back to the base image,
   making `is_prebuilt = false`, so `provision_packages()` runs on every new
   container — each triggering apt-get update + apt-get install (dpkg processes).
4. None of these operations log their output on success, so the user sees dpkg
   processes in htop with zero explanation in moltis logs.

## Changes

### 1. Add logging to `ensure_ready` decision points
**File**: `crates/tools/src/sandbox/docker.rs` (~line 309-373)

- Add `debug!` when container is already running (early return at line 320-322)
- Add `info!` when starting a new container (before docker run at line 352)
- Add `debug!` for the `is_prebuilt` decision (line 368)

### 2. Log docker build output
**File**: `crates/tools/src/sandbox/docker.rs` (~line 407-431)

- On success: log stdout at `debug!` level (truncated to last 20 lines)
- On failure: log both stdout and stderr at `warn!` level (currently only logs stderr)

### 3. Log provision_packages output and track completion
**File**: `crates/tools/src/sandbox/host.rs` (~line 13-40)

- Remove `2>&1 | tail -5` — capture full output
- Log stdout at `debug!` level on success (truncated to last 20 lines)
- Log the actual exit code on failure (not just stderr)

### 4. Add provisioning guard to avoid repeated apt-get runs
**File**: `crates/tools/src/sandbox/docker.rs`

- Track which container names have already been provisioned in a
  `HashSet<String>` on `DockerSandbox` (behind an async `Mutex`).
- In `ensure_ready`, after `provision_packages` succeeds, insert the container
  name. Skip provisioning if the name is already in the set.
- Clear the entry on `cleanup()` so re-provisioning works after container
  recreation.

This prevents the same container from being re-provisioned if `ensure_ready` is
called multiple times before the container dies (the normal idempotent path).

### 5. Log resolve_local_image decisions
**File**: `crates/tools/src/sandbox/docker.rs` (~line 267-300)

- Add `debug!` when image is found locally (early return at line 268-270)
- Existing logs for missing/rebuild paths are adequate
